# Requests for Comments

RFCs record changes that materially affect public contracts, protocol
semantics, privacy, extension APIs, breaking CLI/config behavior, telemetry,
governance, licensing, or trademarks.

## Accepted for v0.1.0 Doctor

- [RFC-0001: Doctor compatibility observations](0001-compatibility-layers.md)
- [RFC-0002: Local evidence and result contracts](0002-evidence-and-result-schema.md)

Acceptance is limited to the explicit Doctor release boundary in each RFC. It
does not stabilize adjacent experimental packages or schemas.

## Deferred ecosystem proposals

- [RFC-0003: Scenario and pack model](0003-scenario-and-pack-model.md)
- [RFC-0004: Driver process and isolation](0004-driver-isolation.md)
- [RFC-0005: Registry trust and attestation](0005-registry-trust.md)

Deferred RFCs are design inputs, not product support, compatibility, service,
or deployment claims. They may change before acceptance.

RFCs use `draft`, `accepted`, `deferred`, `rejected`, or `superseded`. Authors
state the problem, scope, compatibility and security impact, alternatives,
unresolved questions, promotion evidence, decider, and date. Review follows
[GOVERNANCE.md](../GOVERNANCE.md).
