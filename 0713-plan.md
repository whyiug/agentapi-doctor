# AgentAPI Doctor 0713 精简执行计划

> 调研快照：2026-07-13（Asia/Shanghai）
> 目标：给社区一个容易安装、能解释兼容性失败、证据可分享的工具，并用真实案例建立项目和作者的影响力。
> 原则：先交付一个完整用户闭环，再扩协议、SDK、Registry 或 Web；没有证据支持的功能不做。

## 0. 决策摘要

AgentAPI Doctor 解决的是真问题：很多端点能返回 `200 OK`，甚至能通过一条
`curl`，但在 SSE 终态、tool calling、Responses event 或严格 SDK 中仍然失败。
本轮定向调研人工筛选并去重了 18 个公开 GitHub 案例，问题横跨 vLLM、
llama.cpp、Ollama、SGLang、LiteLLM、Open WebUI、LangChain/LangGraph、
Vercel AI SDK、One API、New API、OpenAI SDK 和 Codex。

但调研也否定了两个过大的假设：

- 大众用户通常只想尽快知道 key/endpoint 能不能用，网页 checker、curl 或
  `pipx` 已经更方便；当前没有证据支持再做一个聊天 playground。
- Registry、公开兼容性矩阵、Hosted Web 和同时覆盖大量真实 client 的成本很高，
  当前没有采用数据证明它们值得优先投入。

因此产品只走一条主线：

> **Quick Check 负责让用户两分钟内得到结果；一个真实 client 案例负责证明
> Doctor 比 curl 多解决了什么；可审阅证据负责让结果进入 issue 和 CI。**

核心用户按优先级为：

1. LLM gateway / inference runtime 维护者；
2. Agent、SDK 集成工程师和 platform/CI owner；
3. 遇到“curl 能通、应用失败”的高级 self-host 用户。

大众 key/relay 检查是获客入口，不是产品护城河。

## 1. 调研结论与证据边界

### 1.1 需求是真实的

重复出现的故障主要集中在：

- SSE/event 生命周期：终态缺失、重复、终态后仍有语义数据、代理 buffering；
- tool calling：错误的 `finish_reason`、tool index、空 arguments 或 delta 组装；
- Responses 严格 schema：缺字段、event 顺序或 accumulator 失败；
- adapter/client 差异：直连正常，经 proxy 或真实 SDK/Agent 状态机失败。

代表性公开证据：

- [vLLM #39584](https://github.com/vllm-project/vllm/issues/39584)：Responses
  并行 tool delta 合并导致断流；
- [Open WebUI #21768](https://github.com/open-webui/open-webui/issues/21768)：
  tool call 已发出，但错误终态让 Agent loop 提前结束；
- [llama.cpp #20607](https://github.com/ggml-org/llama.cpp/issues/20607)：
  raw SSE 看似可用，严格 Vercel AI SDK 丢弃 event；
- [LiteLLM #30762](https://github.com/BerriAI/litellm/issues/30762)：普通 stream
  正常，streaming tool call 在 adapter 中丢失；
- [Ollama #15457](https://github.com/ollama/ollama/issues/15457)：多个 tool call
  的 index 错误，需要 proxy 重写；
- [Codex #24973](https://github.com/openai/codex/issues/24973)：普通 curl 和精确
  request replay 成功，真实 client 仍失败。

用户今天通常靠 stream/non-stream、直连/proxy、curl/SDK 的手工 A/B，保存 raw
SSE，修改 provider prefix，编写 70–80 行 wrapper，或直接 patch package 来定位。
这证明专业用户愿意为故障排查投入时间，也说明他们不应再为安装检测器付出额外
成本。

### 1.2 竞品已经占据“测通不通”

| 工具 | 低门槛入口 | 强项 | Doctor 应避开的正面竞争 |
| --- | --- | --- | --- |
| [openai-tests](https://github.com/donadiosolutions/openai-tests) | `pipx run` | OpenAI surface smoke | 只比较请求是否成功 |
| [openai-api-tester](https://github.com/davlgd/openai-api-tester) | Hosted Web / `pipx` | 可编辑请求与 JSON 查看 | 再做一个网页请求器 |
| [api-check](https://github.com/october-coder/api-check) | Hosted Web | model、延迟、可分享结果 | relay/model playground |
| [Am I OpenAI Compatible?](https://github.com/heiervang-technologies/am-i-openai-compatible) | Python / Action | OpenAI 广覆盖与 matrix 雏形 | 只靠 catalog 数量或“有 CI”竞争 |
| [Promptfoo](https://github.com/promptfoo/promptfoo) | `npx` / Web / CI | 通用 eval、报告、历史比较 | 通用模型评测平台 |

Doctor 当前可守住的价值只有：三种 API shape 的 bounded lifecycle 检查、
secret-redacted 内容寻址证据、稳定退出码/报告/baseline。真正的差异化仍需用一个
固定版本真实 client 案例兑现，不能提前宣传为自动根因定位。

### 1.3 不能从这轮调研推出什么

这是 purposive search（定向检索），不是随机抽样或市场规模研究。它证明问题
跨项目反复发生，但不能证明 TAM、付费意愿、故障率或大众需求规模。Issue 关闭
也不自动等于已修复。项目不把 star、catalog 数量或本仓 CI 绿灯当作采用证明。

### 2.1 Reasoning-aware Quick Check 预算契约

`v0.1.0-rc.2` 的每请求固定 64-token 请求值对结构检查足够，但不是跨生态中立的
terminal-status 前置条件。OpenAI 的 completion/output 上限包含 reasoning tokens，
Anthropic thinking 与最终文本共享生成上限，DeepSeek 当前 thinking 默认启用；字段
名字相同或相似也不代表各 Provider 的计费与截断语义完全一致。设计依据见
[OpenAI reasoning guide](https://developers.openai.com/api/docs/guides/reasoning)、
[Anthropic extended thinking](https://platform.claude.com/docs/en/build-with-claude/extended-thinking)、
[DeepSeek thinking mode](https://api-docs.deepseek.com/guides/thinking_mode/) 和
[火山方舟 Chat API](https://www.volcengine.com/docs/82379/1494384?lang=zh)。

在用户授权的火山方舟 Seed 2.0 Lite endpoint 上，以合成 prompt 做过一次有界 A/B：
默认 thinking 下 64、128、256 均以 output limit 结束，512 在该次调用中自然结束；
禁用 thinking 的 64-token 调用也自然结束。该观察只支持调整有界默认值，不支持
对任意 Provider、model 或长期稳定性作认证。rc.3 候选随后用同一配置执行完整
4-request Quick Check，得到 4 PASS / exit 0；临时 binary、HOME、evidence 和输出已
清理，该单次回归仍不构成跨 Provider 兼容声明。

因此 rc.3 采用以下版本化契约：

- 12 个内置场景逐一绑定 requested output tokens；只有 OpenAI Chat 和 Responses 的
  terminal-status 场景使用 512，其余结构、envelope、stream lifecycle 场景保持 64；
- 每次仍固定 4 请求、串行、零重试；Chat/Responses 的 requested maximum 为 704，
  Anthropic Messages 为 256，并在网络前写入 plan reservation；
- 不注入 Provider-specific thinking 开关，不按 model 名称建立 allowlist，也不在
  命中上限后用隐藏请求自动扩容；
- Provider 明确报告命中请求上限时仍为 `INCONCLUSIVE`，reasoning-token usage 只增强
  expected/observed 诊断；usage 未返回时 attempt 与持久化 report condition 记录
  unknown，不伪装为实测 0；output usage 未知时 ledger 用该请求的 reservation 保守
  记账，input usage 因没有 Provider-neutral estimate 而保持 unknown；完成响应若使
  累计账务超过批准的 run budget，保留 target verdict，但整体必须为
  `INCONCLUSIVE` / exit 4；单请求超过 estimate 但累计值仍在 run budget 内时只记录
  reservation overshoot，不改写成 Provider incompatibility；
- built-in identity 升为 `0.1.0-candidate.3`。旧记录保持可读，但 rc.2 baseline 与
  rc.3 pack/profile/plan identity 不可比较，必须重新采集而不是改写 digest；
- 512 是当前有证据支持的有界默认值，不保证所有 reasoning model 自然结束；后续只
  在获得授权的代表性 Provider canary 证据后调整 profile，不扩大本轮产品声明。

## 3. 三阶段执行主线

### P1 — 让陌生用户下载后立即看懂结果

目标：不安装 Go、不 clone 仓库，两分钟内运行 demo 或第一次 Quick Check。

本阶段交付：

- [x] FAIL/WARN/INCONCLUSIVE 输出人类可读的检查名和 expected/observed；证据支持
  finding 时显示 fault domain/remediation，否则明确未归因并给出 next action；
  PASS 保持紧凑；
- [x] `doctor help`、`doctor test --help`、`doctor demo --help` 和
  `doctor report --help` 正常输出并返回 0；
- [x] 准备 `v0.1.0-rc.1` release notes 和 known limitations；
- [x] 首发只构建 `doctor`：Linux、macOS、Windows 的 amd64/arm64 archive，Windows
  使用 ZIP；
- [x] 发布 SHA-256 checksum、一个 SPDX SBOM 和 GitHub build provenance；
- [x] 在 Linux、macOS、Windows 的代表性原生 runner 下载同一批 artifact，校验
  checksum，运行 `version` 与 `demo`；
- [x] 发布后再从公开 Release 下载并 smoke，确认不是只验证 Actions 临时 artifact；
- [x] 中英文 README、Quick Start、Installation 默认展示预编译安装，源码安装降为
  开发者选项；
- [x] 加入一个来自真实合成故障运行的终端视觉和一个可离线打开的 HTML report
  示例。

首发**不再等待** Registry/reference-server binary、OCI image、Homebrew tap、
Scoop bucket、GitHub Action 或 reusable workflow。已有实现可保留为 unpublished
candidate，但 README 不把它们写成可用渠道。

验收门：三个主流 OS 的公开下载路径都能完成 `version` 和 `demo`；失败输出能让
第一次使用者说出“哪里不符合预期、下一步做什么”。

### P2 — 用一个真实 client 证明不可替代价值

目标：完成一个可重跑的 `raw wire → pinned SDK` 纵切，不先扩三个生态。

首个 profile 固定为 **OpenAI Python SDK / Responses streaming**。只覆盖两个最
典型 failure family：

1. missing/duplicate terminal event；
2. strict Responses event/envelope field 缺失或不一致。

交付：

- [x] 锁定 CPython 3.12.12、OpenAI Python 2.38.0、SDK tag commit、wheel SHA-256、
  完整 wheel-only dependency lock 和 fixture provenance；
- [x] 从本计划调研到的公开 failure family 独立实现许可证清晰的合成 mutant；不复制
  issue payload、上游测试或生产数据；
- [x] 同一次运行分别记录 raw wire 与 SDK observation；
- [x] 证据不足时输出 `UNKNOWN`，不把 client exception 自动归咎于 provider；
- [x] 导出 maintainer-ready bundle：summary、版本、运行命令、expected/actual、
  脱敏 evidence、client error 和修复后回归步骤；
- [x] 在 Ubuntu clean runner 从 hash lock 构建临时 wheelhouse，离线安装依赖，把
  reference 和三个 targeted mutant 各运行两次；另有单测验证相同输入生成
  byte-identical bundle；该 job 纳入 `Product CI / aggregate`。

初版不强制加入 proxy 层或 Agent loop；只有目标案例确实涉及它们时才增加。
Vercel AI SDK 是第二 profile 候选，不是预先承诺。

验收门：至少一个公开 failure family 能经固定版本真实 SDK 确定性复现，且输出
比手工贴 raw log 更容易被维护者复跑和审阅。

### P3 — 把真实闭环变成采用与传播

目标：让项目获得外部复用，而不是只在自己的仓库中证明自己。

交付：

- [x] 发布一篇
  [可复跑案例](https://github.com/whyiug/agentapi-doctor/blob/v0.1.0-rc.2/docs/cases/openai-python-responses-null-output.md)：
  普通 smoke 为什么漏掉、Doctor 观察到什么、真实 SDK 如何失败、维护者如何复现；
- [x] 发布可离线打开的 HTML failure 示例和 clean-runner 最小 CI snippet；
- [ ] 发布 20–40 秒合成 failure demo；
- [ ] 邀请至少 5 位非作者用户对获授权 endpoint 完成一次真实 run；
- [ ] 至少 2 位用户实际分享一份人工审阅的脱敏 report、接受一个 repro bundle，
  或固定版本接入一次 CI；口头“以后会用”不计；
- [ ] 根据失败原因和用户行为决定第二个 client profile，而不是按预设 roadmap
  扩张。

渠道以事实资产为中心：GitHub Release、README、Topics、Hacker News Show HN、
Reddit LocalLLaMA/self-hosted 和相关中英文社区。只在能给上游 issue 提供最小
repro、fixture 或修复验证时参与，不把 issue 当广告位。

验收门：至少一个外部用户或仓库固定版本复用 Doctor，并产生可核查的 report、
bundle 或 CI 结果。

## 4. 只有证据出现后才做的 backlog

- 第二个真实 client profile；
- 长期 reusable workflow / GitHub Action 产品化；
- 多 design partner 和 4–6 周 drift pilot；
- 版本绑定的社区 compatibility matrix；
- 公共 Registry；
- local Web UI 或 hosted checker；
- Homebrew tap、Scoop bucket 和其他包管理渠道。

触发规则：

- 如果 Quick Check 有使用、真实 client/CI 无转化，保留轻量 CLI，不扩 evidence
  platform；
- 如果 maintainer/CI 复用明显，优先做第二 client、drift 与 issue-ready bundle；
- 如果两者都弱，收缩为 targeted fixture/conformance library，考虑向已有项目
  贡献检查；
- Matrix 只有在至少 20 个经同意、版本绑定、可重跑的独立 observation 出现后
  才重新评估；
- Web 只有在真实用户证明“安装仍是主要流失原因”后才重新评估。

## 5. 推广与衡量

统一信息：

> **200 OK ≠ compatibility. Check the stream. Keep the evidence.**
> **200 OK 不等于兼容。检查协议流，把证据留下来。**

推广资产按可信度递进：

1. RC binary、无 key demo、合成 lifecycle failure、离线 HTML；
2. 固定版本真实 SDK failure 和 maintainer-ready bundle；
3. 经同意的外部 CI 或上游接受案例。

即使 P2 完成，也不宣传“自动定位根因”“完整 SDK 兼容”或“社区兼容矩阵”。所有
图片、文章和 release notes 都必须来自可重跑输出，不编造 PR、issue、adopter 或
评价。

只跟踪四个有行动意义的信号：

- 首次安装到可解释结果的完成情况；
- 独立非作者真实用户数；
- 外部仓库或报告中的固定版本引用；
- 被上游接受的 fixture、repro 或修复验证数。

下载量和 star 只作为传播信号，并注明 bot、CI 重复下载和缓存等限制。

## 6. 安全与诚信护栏

- 只测试用户明确授权的本地、私有网络或远程 endpoint；不扫描、探测或 fuzz
  未授权公网目标；
- credential 只通过 env/file secret reference 读取，不进入命令行、日志、报告或
  repository；
- 默认本地保存和脱敏；分享前仍要求人工审阅，因为 model content/tool arguments
  可能不是匿名数据；
- 保持 request/deadline/token reservation；Provider 可能拒绝或忽略 64/512-token
  请求字段，因此不把它们描述成客户端强制成本上限；
- 真实 client 与 fixture 固定版本；旧观察不自动代表 latest；
- 证据只支持 observation，不支持的 fault domain 必须标成 `UNKNOWN`；
- 不做模型质量 benchmark、价格/额度排行、vendor certification 或通用聊天
  playground；
- 不用 LLM 判断替代 wire evidence、真实 client reproduction 和人工校验。
